版本 1.13.7 · Windows · Electron

从脚本执行到 Typora 持续运行，整个过程是一条单向流水线。下图展示从启动到稳定运行的全生命周期。

图0 —— 全链路总览

一、Typora 的激活验证架构

Typora 的 License 验证并非单一检查点，而是一条三层验证链路。任何一层失败，Typora 都会判定 License 无效并回退到试用模式。

第一层：本地 License 解密

Typora 启动时读取本地存储的 License 文件（加密的 JSON），通过 Node.js crypto.publicDecrypt 进行 RSA 解密，得到包含设备 ID、邮箱、到期日期的 License Entity。如果解密失败或日期已过期，直接进入试用模式。

第二层：UI 联网校验

渲染进程（UI 窗口）在启动后向 https://api/client/activate 发起 HTTPS 请求，将本地 License 信息提交到 Typora 服务器进行二次确认。服务器返回的响应中包含签名的 StandardLicenseMsg（base64 编码），UI 据此显示"已激活"或"未激活"。

第三层：后台定时续订

主进程通过 electron-fetch 模块，每 12 小时自动向 api/client/renew 发送续订请求。Typora 的 JavaScript 核心驱动这个定时任务。如果续订请求到达真实服务器且返回失败，License 立即被撤销。

结论：一个完整的激活方案必须在全部三个验证层上进行拦截，且由于主进程的网络请求存在两个不同的底层通道（net.fetch 和 net.request），第三层还需要双通道覆盖。

二、代码注入与文件隔离

所有拦截逻辑需要在 Typora 的 JavaScript 执行环境中最先运行。这涉及两个技术前提：代码注入和文件隔离。

注入载体

Typora 的入口文件是 resources/app/launch.dist.js。该文件在 Electron 启动流程中最早被执行，在其开头插入拦截代码可以确保 Hook 在任何 Typora 业务逻辑之前生效。但 Typora 的代码以 app.asar 格式打包——这是一个只读的归档文件，无法直接修改。因此需要先用 asar.extractAll 解包为普通文件夹 app/，再对 launch.dist.js 进行修改。

文件系统隔离

修改 launch.dist.js 后，其余数百个原始模块需要与注入代码隔离，防止冲突。将 app/ 完整复制为 app.bak/，再通过路径重定向让 Typora 从 app.bak/ 加载非注入模块。结果：launch.dist.js 从 app/ 加载（含 Hook），其余模块从 app.bak/ 加载，二者物理隔离、逻辑共存。

四、网络请求通道与拦截决策

Typora 向 api/client/ 发起的请求存在三种不同的底层通道，取决于发起方（渲染进程 / 主进程）和使用的模块（原生 fetch / electron-fetch）。每种通道需要独立的拦截层。

五、各拦截层的实现机制

层 1 文件系统重定向

Typora 在启动后会通过 Node.js fs 模块加载大量 JS 模块。拦截 fs.readFileSync、fs.readFile、fs.stat、fs.statSync、fs.open、fs.openSync 以及对应的 Promise 版本，在路径参数匹配 resources\app\ 时替换为 resources\app.bak\。

const fsPathFrom = /resources[\\/]app[\\/]/i;
const fsPathTo = "resources\\app.bak\\";
// fs.readFileSync, fs.readFile, fs.stat, fs.statSync,
// fs.open, fs.openSync, fs.promises.* 全部拦截

这层拦截确保了：注入代码（位于 app/）和原始业务代码（位于 app.bak/）完全物理隔离，互不干扰。

层 2 crypto 加解密拦截

Typora 使用 Node.js crypto.publicDecrypt 解密本地 License 文件。拦截此函数后，不再执行实际的 RSA 运算，直接构造一个合法的 License Entity JSON 并返回。

crypto.publicDecrypt = function(key, buffer) {
    const entity = { deviceId, fingerprint, email, license, version, type };
    entity.date = (() => {
        const d = new Date();
        return (d.getMonth()+1)+'/'+d.getDate()+'/'+d.getFullYear();
    })();
    return Buffer.from(JSON.stringify(entity));
};

同时覆盖 crypto.privateDecrypt，因为不同版本可能使用不同的解密 API。

层 3 渲染进程协议拦截

Electron 的 protocol.handle("https", ...) 可以拦截渲染进程中所有通过 Chromium 网络栈发出的 HTTPS 请求。匹配 URL 中包含 api/client/ 的请求，返回一个动态生成的 StandardLicenseMsg（base64 编码）。

覆盖范围：仅渲染进程。主进程的 electron.net.fetch 和 electron.net.request 绕过此层。

层 4 主进程 net.fetch 拦截

拦截 electron.net.fetch，这是 Electron 提供的 Promise 风格网络 API。主进程中直接调用此 API 的场景由此层覆盖。

覆盖范围：主进程中使用原生 net.fetch 的请求。不覆盖 electron-fetch 模块。

层 5 主进程 net.request 拦截

最关键的一层。Typora 的 License 续订由 electron-fetch 模块驱动，该模块在 Electron 环境中的底层实现是 electron.net.request——一个返回 ClientRequest 对象的 EventEmitter 风格 API，与 net.fetch 的 Promise 风格完全不同。

const origNetRequest = electron.net.request.bind(electron.net);
electron.net.request = function(options, callback) {
    if (url && url.includes('api/client/')) {
        const mockReq = new EventEmitter();
        mockReq.end = function() {
            mockReq.emit('response', mockRes);
            mockRes.emit('data', Buffer.from(resBody));
            mockRes.emit('end');
            return mockReq;
        };
        return mockReq;
    }
    return origNetRequest(options, callback);
};

拦截逻辑模拟了 ClientRequest 的完整事件流（request → response → data → end），使 electron-fetch 的调用方如同收到了真实服务器的响应。关于 net.fetch 与 net.request 为何是两条独立通道、为什么拦截层 4 无法覆盖此场景，详见第六章。

层 6 试用数据清理

Typora 的试用状态持久化在 Chromium 的多种存储中。即使 License 验证被拦截，如果残留的试用数据未被清除，UI 仍可能显示"剩余 X 天试用期"。

数据类型具体项目
Chromium 存储Local Storage、Session Storage、DIPS、Code Cache、GPUCache、Network、SharedStorage
Windows 注册表SLicense、IDate、TrialStart、TDate、FirstLaunch、LaunchCount

六、electron-fetch 的底层通道问题

electron-fetch 是一个第三方 npm 包，Typora 将其打包在 node_modules.asar 中。它的核心路由逻辑为：

// electron-fetch/lib/index.js
const send = request.useElectronNet
    ? electron.net.request        // Electron 环境 → 走这里
    : (options.protocol === 'https:' ? https : http).request;

其中 useElectronNet 的定义是 Boolean(process.versions.electron)。在 Electron 进程中，此值恒为 true，因此 electron-fetch 永远选择 electron.net.request 作为底层传输。

这与 electron.net.fetch 是完全不同的 API：

net.fetchnet.request
返回值Promise<Response>ClientRequest (EventEmitter)
调用方主进程直接调用electron-fetch 间接调用
拦截层层 4层 5

这一机制解释了为什么必须在 net.fetch 拦截之外，独立地实现对 net.request 的拦截——两个 API 没有继承关系，也不共享调用路径。

七、License 日期的动态绑定

License Entity 中的 date 字段不可硬编码为某个固定日期。原因是 Typora 在每次验证时会比较当前系统时间与 License 中的到期日期——如果 License 日期是过去的某一天，验证将失败。

因此，所有拦截层在构造 License Entity 时均使用 运行时动态计算的当天日期：

entity.date = (()=>{
    const d=new Date();
    return (d.getMonth()+1).padStart(2,'0')+'/'+
           d.getDate().padStart(2,'0')+'/'+
           d.getFullYear();
})();

这段逻辑被复制到拦截层 2、3、4、5 共四个拦截器中，确保无论 Typora 从哪个通道读取 License 信息，得到的 date 都等于"今天"。

八、完整验证链路闭合

将各拦截层串联，Typora 从启动到持续运行的全生命周期中，License 验证链路如下：

Typora 启动 → 层 1 文件系统重定向，模块从 app.bak/ 加载
→ 层 6 清除残留试用数据
→ Typora 读取并解密本地 License → 层 2 伪造解密结果（签名校验被绕过）
→ UI 窗口联网校验激活状态 → 层 3 返回假响应 → 界面显示 "已激活"
→ 12 小时后主进程自动续订 → 层 5 拦截 electron-fetch → 续订成功 ✓

三层验证、三条通道、六个拦截层——每一层覆盖一个独立的验证路径，不存在单点故障。Typora 在任何时刻、通过任何通道检查 License 状态，拿到的都是动态生成的"有效"结果。
