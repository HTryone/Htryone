一、概述

Typora 是一款基于 Electron 框架构建的 Markdown 编辑器。Electron 应用的核心代码打包在 resources/app.asar 文件中。激活脚本通过解包 → 注入 Hook → 重打包的方式， 在 Typora 的 JavaScript 运行时中植入拦截代码，从源头欺骗许可证验证的每一个环节。

核心思想：不要尝试"算出"一个合法的序列号。而是劫持验证逻辑， 让 Typora 不管发送什么验证请求，都会收到一个"已验证通过"的假响应。

关键操作详解

  1. 解包 app.asar
    使用 asar.extractAll(asarPath, appDir) 将压缩归档解包为普通文件夹。 Typora 的所有 JS 代码现在以明文形式存在于 resources/app/ 目录下。
  2. 备份原始代码
    将 resources/app/ 整体复制到 resources/app.bak/。 后续 fs Hook 会让 Typora 读取 app.bak/ 中的原始文件，确保除注入代码外，一切逻辑不变。
  3. 备份原始 asar
    app.asar 重命名为 app.asar.bak，用于回滚还原。
  4. 修改 Electron Fuses
    使用 @electron/fuses 的 flipFuses 将 Typora.exe 中的 OnlyLoadAppFromAsar 从 true 改为 false， 使 Electron 允许从解包的 app/ 目录加载代码。
  5. 注入 Hook 代码
    在 app/launch.dist.js 的第一个 require(...); 语句之后插入 Hook 代码。 launch.dist.js 是 Typora 的入口文件，在此注入可确保 Hook 在所有业务代码之前运行。
  6. 写入注册表
    向 HKCU\Software\Typora 写入 SLicense 和 IDate， 作为静态授权凭证。Typora 启动时检测到这些键值会认为"已经激活过"。

各层 Hook 详细说明

# Hook 层 拦截目标 运行时机 优先级
1 fs 路径重定向 fs.readFileSync、fs.readFile、fs.stat、fs.open 等 8 个同步方法 + fs.promises 的 3 个异步方法 脚本加载时立即执行 极高

6 electron.net.fetch 主进程其他 net.fetch 调用，作为第5层的后备补充 app.whenReady() 回调 低

四、网络请求拦截 — 最复杂也最关键的部分

⚠️ 为什么需要 3 层网络拦截？
Typora 的许可证验证涉及 两个进程 + 两个网络库，单一 Hook 无法覆盖所有请求路径。 这是经历多次"激活几天后又失效"的教训后得出的结论。

以防万一：date 动态生成机制

所有 Hook 返回的 entity.date 不是硬编码值，而是在每次调用时实时计算： 

entity.date = (() => {
  const d = new Date();
  return String(d.getMonth()+1).padStart(2,'0') + '/' +
         String(d.getDate()).padStart(2,'0') + '/' +
         d.getFullYear();
})();
为什么必须动态？
2026-06-01 曾出现"激活后第二天就失效"的问题。原因是 msg 中的 date 在激活时静态生成为 base64，第二天 Typora 检测日期不一致就认为过期。
修复方案：所有 Hook 回调中 不存静态值，运行时动态生成 ——每次请求都用当天日期。

七、关键修复历史

激活脚本不是一次就完美的，以下是经过 3 次重大修复才达到当前稳定状态的过程：

2026-06-01
🐛 永久激活第二天就失效
现象：激活当天正常，次日 Typora 又回到未激活状态。
根因：msg 中的 date 使用激活时静态生成的 base64，
第二天日期不匹配导致验证失败。
修复：protocol.handle 回调改为运行时动态生成 entity + msg。
2026-07-01
🐛 激活后显示"还有12天体验期"
现象：虽然部分 Hook 生效，但 Typora 仍显示试用倒计时。
根因：① protocol.handle 只能拦截渲染进程，主进程 net.fetch 不受控； ② 试用数据持久化在 Local Storage/DIPS 中独立运行倒计时； ③ URL 匹配范围太窄，只拦截 activate/renew。
修复：新增 net.fetch Hook + 扩展 URL 匹配到全部 api/client/ + 新增试用数据清理函数。
2026-07-13
🐛 试用期又提示"3天到期" ⭐ 最终根因
现象：7月1日修复后，13天后又提示试用到期。
根因：Typora 使用 electron-fetch 模块（存放在 node_modules.asar）， 该模块在 Electron 环境默认 useElectronNet=true， 调用的是 electron.net.request（不是 fetch！）。 net.fetch Hook 完全无法拦截。每12小时续订请求穿透到真实服务器 → 服务器返回失败 → Typora 撤销 license → 进入试用模式。
修复：新增同步 electron.net.request Hook， 构造完整的 fake ClientRequest + response → data → end 事件流。 这是最关键的一次修复。
经验教训：Electron 环境中的网络请求存在多种实现路径。 仅拦截 fetch 和 protocol 是不够的， 必须深入分析应用实际使用的网络库（如 electron-fetch）并拦截其底层调用。
